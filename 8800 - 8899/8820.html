抱歉这篇issue不能按google code 指定的格式发言。因为我想提一个建议而不是报告一个技术问题。
goagent真的很好用，最赞的是速度。但是由于gae平台的限制，下载大文件易出错，而且由于gae分配的ip不停变化，无法浏览检查ip恒定性的视频网站和或从网盘下载。

另一款优秀的利用云平台的代理软件是apjp，在gae平台上，由于apjp没有提供多ip和ipv6支持，对国内网络环境的适应不如goagent好，可是由于它支持jelastic/heroku/cloudcontrol/php..../很多，还有更好的安全性，对于gae平台做不到的，如我提到的网盘下载等，其它的平台能够做到。

可利用其它云平台产生的问题是国内连接的速度，电信这方面要比联通等好得多，可是gae以外的平台(由于大部分利用aws美东的数据中心）还是在速度上没法跟gae比。

apjp已经能做到用一个paas当另一个paas的代理，如me→jelastic→openshift→internet, 我的jelastic选的日本服务器，速度相对快，openshift为美国固定ip，我用北京联通6m adsl连openshift只有50k/s的速度，连jelastic-japan 300k/s,这样连接的优势第一是突破了一些网站只允许美国ip登录，第二又让我连接openshift获得了固定ip并且速度是jelastic的速度。

由于我前阵在一个偏僻地方用铁通2m网络，现在在用北京联通，这两个isp下，gae都是满速，因此最理想的方式是用gae做其它paas的前置代理。但由于gae有url fetch 小于5m，url response 小于 32m等等种种限制，虽然可以修改将大文件分割成小文件依次下载，其它paas的remote server 配置文件需要改变以适应gae。

我想gae平台上做得最好的，尤其是速度方面，是goagent，由于apjp作者已参与 issue 160 讨论，goagent作者能否进入讨论给apjp作者一些技术上的建议，甚至你们俩可以合作。毕竟gae的ip无限变化造成的问题太大了。而apjp虽然适应广泛的云平台，但由于其作者不懂python，使用java的apjp对系统资源占用不小，而且无法在ios平台上应用。

issue: https://code.google.com/p/apjp/issues/detail?id=160
